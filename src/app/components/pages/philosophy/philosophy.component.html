Philosophy of C++

One of the most common mistakes people make with C++ is not taking the time to read up on the lessons learned previous to the inception of the language, such the reasons why the language came into being in the first place. From reading his book, The C++ Programming Language, I do remember Bjarne explicitly mentioning how writing too much code is essentially writing too much code. People that happen to think that they call solve all the world's problems away merely by writing a lot of extra code (just in case) usually end up producing a monster that neither addresses the original problem nor is easy to maintain, much less extend. Like a martial artist who founds his own martial art based on his own experiences with previously established martial arts, so also it is a very wise thing to do to study what the inventor of a particular programming tool had in mind before embarking on a journey. Unless making the same series of mistakes that led to the invention of new programming tool is the preferred learning style of the student in question.

#1 Make code readable and logical in it's layout

It would take a course in Object-Oriented Design to truly appreciate what C++ offers in terms of writing code that reflects the real world object it is meant to simulate in a programming language. Essentially, you are going to describe a real world object in terms of type or class. Further, all attributes related to that class can be represented in code as an attribute and all functionality related to the real world object can be represented in code as a method. These attributes and methods can then be expressed in code inside the class itself, usually with some form of indentation so that the ideas the code is meant to convey is immediately relatable to the person reading the code. In this way, the C++ code you write becomes very intuitive, that is, you can quickly see what the code represents in the real world as well as it's characteristics.

class Person {{"{"}}
// attibutes
String name;
Age age; // do
int age; // don't
public:
// methods
Route walk() const; // do
Route walk(); // don't
{{"}"}}

By default a person's age can be represented using an int value. This is perfectly logical. But what if we wanted to ensure that the person's age never has a negative value, what then? One way to ensure that a person's age never has a negative value would be to have a class dedicated to managing the characteristics of an Age as far as the system is concerned. Hence you would write a dedicated class for Age:

class Age {{"{"}}
// attibutes
int age;
public:
// methods
Age(int age) : age(age)
{{"{"}}
if (age<0)
throw new IllegalAgeException();
{{"{"}}
{{"}"}}

Further the 'const' keyword is used on the method walk(). This signifies that the method when called will not change the object in anyway, just return a Route object that can be used by the calling application. To a programmer new to this code, knowing that calling the walk() method on this object without having to worry about a changed state of the object after the method is called can be very important, especially in a real world environment. In effect, forgetting to include the 'const' keyword raises an unnecessary complication when utilizing this code for future projects.

#2: Write your core code using ISO Standard C++

Every major vendor of C++ has their own unique way of doing things, especially when it comes to interfacing with the local GUI and databases. The core logic of application, the very purpose of your application should be independent of this. In this way, you will be able to port the core portion of your code to another platform with little or no difficulty. By 'core' I mean the brains of your application, that part of the code that the project set out to accomplish that could not be readily purchased off the shelf. A popular design pattern to support this practice is known as MVC or Model View Control, where the View and the Control aspect of your software tends to be very specific to the local operating system, but the brains of your application does not need to be tied into your local operating system at all if you limit it to using ISO Standard C++ only:

An example of ISO Standard C++ application would be one that by default interfaces with the console only. Examples of a non-standard C++ application would be like Apple's Xcode and Microsoft's Visual C++. Both popular platforms for C++ development but also completely non portable to each other (without a lot of effort). The idea is that you should write your core logic using only the console as your input and your output and then interface with either Xcode or Visual C++ using a MVC design framework:

Further setting up a good test suite to ensure that your core code is functioning as expected is a good way to keep your code portable and a really good way to ensure that your code works as expected when ported. In effect, your test code acts as a kind of view in the MVC paradigm.

#3 Write code that's intuitive and pleasant to the eye

Taking the time to make your finished product more readable can go a long way as far as code re-use is concerned. It also makes for good code readability. Rather than just using a single letter to express a given purpose, use a logical identifier which best describes what the code is actually doing. Having this mentioned in comments is one thing, but often times comments fall out of sync with what the final product actually did to accomplish the goal.

// With poorly descriptive identifiers
struct yx {{"{"}}
std::string name;
{{"}"}} x[] = {{"{"}} {{"{"}} "Archimedes" {{"}"}}, {{"{"}} "Vitruvius" {{"}"}}, {{"{"}} "Teka" {{"}"}}, {{"{"}} "Zepo" {{"}"}} {{"}"}} ;
int y = 4;
for ( int i = 0; i<y; i++ ) {{"{"}}
std::cout << "The dancing bear's name is " << x[i].name << std::endl;
{{"}"}}
// With better descriptive identifiers
struct bear {{"{"}}
std::string name;
{{"}"}} bears[] = {{"{"}} {{"{"}} "Archimedes" {{"}"}}, {{"{"}} "Vitruvius" {{"}"}}, {{"{"}} "Teka" {{"}"}}, {{"{"}} "Zepo" {{"}"}} {{"}"}} ;
int bear_count = sizeof(bears)/sizeof(bear);
for ( int index = 0; index<bear_count; index++ ) {{"{"}}
std::cout << "The dancing bear's name is " << bears[index].name << std::endl;
{{"}"}}

The first code block just uses one or two letters to describe important concepts that the solution is modelling in the code. The second code block takes the time to properly spell out what the code block is actually processing. The second code block also demonstrates how an observant programmer can quickly identify a subtle problem in the code such that the value of bear_count is dynamically calculated as oppose to static set in the first code block.

#4: Implement strong type checking when possible

One of the benefits of C++ is it's ability to double check the type of information you are using. Before the days of C++ you could use C to perform any form of struct manipulation on arrays and unions and while this seemed a rather cool way of doing things, they often introduced subtle errors especially when ported to different platforms. One way around this is to take the time set everything up inside of it's own class or typedef. With strong type checking in place, it is possible to minimize the number of subtle problems that can arise with large programs, especially those of a legacy nature. That is, code you have inherited from another project.
Problem areas for type checking are unions, casts, array decay, range errors and narrowing conversions. A cast is one such operation where type checking gets overridden, ideally, one should never have to make use of a cast. But it is not always the case as there are many instances where making use of a cast saves the programmer time in the short term at the possible risk of introducing a subtle bug that is difficult to track down at a later time.

Here we have an abstract class called Animal:

class Animal {{"{"}}
public:
virtual void makeSound() = 0;
virtual int getLegCount() = 0;
{{"}"}};

Here we have two classes derived Animal, Person and Dog respectively:

class Person : public Animal {{"{"}}
std::string name;
public:
Person(std::string name)
:name(name) {{"{"}}{{"}"}}
void makeSound() {{"{"}}
std::cout << "My name is " << name << std::endl;
{{"}"}}
int getLegCount() {{"{"}}
return 2;
{{"}"}}
void gotoWork() {{"{"}}
std::cout << "Deliver Mail\n";
{{"}"}}
{{"}"}}
class Dog : public Animal {{"{"}}
std::string name;
public:
Dog(std::string name, int age)
:name(name) {{"{"}}{{"}"}}
void makeSound() {{"{"}}
std::cout << "My name is " << name << std::endl;
{{"}"}}
int getLegCount() {{"{"}}
return 4;
{{"}"}}
void bitePostman() {{"{"}}
std::cout << "CHOMP!\n";
{{"}"}}
{{"}"}}

The first code block shows a typical abstract class with two virtual functions makeSound() and getLegCount(). Any derived classes have to implement these classes in order to be instantiated as a Animal object.

Person person("Harry"); // creates a Person object for Harry
Dog dog("Lucky"); // creates a Dog object for Lucky
Animal* pAnimal = &person; // Assigns a Animal pointer to Harry
pAnimal->makeSound(); // " My name is Harry " is printed
std::cout << pAnimal->getLegCount() << std::endl; // " 2 " is printed
pAnimal = &dog; // Assigns the same Animal pointer to Lucky
pAnimal->makeSound(); // " BARK BARK Lucky " is printed
std::cout << pAnimal->getLegCount() << std::endl; // " 4 " is printed

The next code block shows what happens when a type safe checking is ignored by the programmer. The program is told explicitly that a dog is a person and tries to carry out operations on the person that is really a dog. When this happens C++ functionality is unpredictable. In this case, it successfully prints out the message reserved for a Person object, no errors are reported, no exceptions are thrown. This is the worse kind of problem to have in a type cast situation. In the following code the method called on the cast is incorrect while the value returned is correct.

Person* pPerson = &person; // Assigns a Person pointer to Harry
pPerson->gotoWork(); // " Deliver Mail " is printed
std::cout << pPerson->getLegCount() // " 2 " is printed
pPerson = (Person*)&dog; // Erroneously cast a Person pointer to Lucky
pPerson->gotoWork(); // Erroneously " Deliver Mail " is printed
std::cout << pPerson->getLegCount() // " 4 " is printed

The next code block reiterates the same issue with the assignment is in reverse. That is, when a pointer to a Dog object is given a Person object instead. If some sort of exception were to be thrown so as to be able to detect this problem all would be fine and good, but in this case, no exceptions are thrown and unless the programmer has a solid test case in place to check for correct values, there is no way of knowing that the cast is pointing to an incorrect object!

Dog* pDog = &dog; // Assigns a Dog pointer to Lucky
pDog->bitePostman(); // " CHOMP! " is printed
std::cout << pDog->getLegCount(); // " 4 " is printed
pDog = (Dog*)&person; // Erroneously cast a Dog pointer to Harry
pDog->bitePostman(); // Erroneously " CHOMP! " is printed
std::cout << pDog->getLegCount(); // " 2 " is printed

The best way to deal with type safe issues is to take the time to setup an effective test suite to test for correct values from every possible intended usage of the object. Having someone on staff that likes to write a lot of seemingly unnecessary code is ideal to be assigned to write test code and to setup and manage test suites and perhaps even to correlate with the requirements management staff on what tests are to be written. But that same person is not someone you want on the architecture team when it comes to finding ways to write less code.
Further, while it can be argued that the abstract method getLegCount() behaved correctly, that is, successfully retrieving the value of the object without any issues, fundamentally the program is incorrect as a Person rarely has 4 legs and vice versa.

#5: Use compile-time validations instead of run-time validations

Ideally, you will setup a test case for everything you do in your code. Such that there is nothing that happens in real time that your test case hasn't run a check for in advance. If you do this, you will be able to detect subtle run-time errors a lot sooner than you would otherwise. That being said, there are ways of checking your code at compile-time which reduce the need to check it at run-time. Where possible try to check your code at compile-time over run-time.

The difference between compile-time and run time is one of the hardest concepts to learn for people without much background in programming languages. To address this problem I find it helpful to ask the following questions:

  1. What parts of the problem to be solved does not change? If any?
  2. What can go wrong with the parts of the problem that do change?
  3. After the program runs, what happens? What changes occurred?
  4. Are there any parameters needed before the program is run?
  5. Is there any input data to be acted upon, will it produce any output?

Use a Compile-time solutions if ...

  1. If the program does not need to change static data.
  2. If the errors to be encountered can be detected at compile time.
  3. If no errors are reported, is the program doing what is suppose to do?
  4. If the data is static, compile it inside the program if possible.

Use a Run-time solutions if ...

  1. If we know nothing about the data to be processed except it's type.
  2. You are prepared to write the required error-handling to handle errors encountered at runtime.
  3. The program needs to be intelligent enough to govern itself.
  4. If the program needs to be able to interpret run-time type information on a dynamic basis (RTTI).

For example, regardless of compiler, you can always save on runtime if you can afford to do ...

if (typeid(a) == typeid(b)) {{"{"}}
B* ba = static_cast(&a);
etc;
{{"}"}};

Instead of ...

B* ba = dynamic_cast(&a);
if (ba) {{"{"}}
etc;
{{"}"}}

The former involves only one comparison of std::type_info; the latter necessarily involves traversing an inheritance tree plus comparisons.

#6: Use run-time validations when compile-time validations aren't available

This boils down to the difference between C programming practices and C++ programming practices. Examine the following code:

extern void x(int* y);
void z(int n)
x(new int[n]);
{{"}"}}

There is no way for the x function to know how large the array being passed to it is. Which is fine as long as the algorithm being implemented is simple and understood to not be an issue when the original solution was implemented. But what if this code becomes production code and an upgrade down the line makes things more complicated, perhaps extends the size of the array such that it throws a nasty null pointer exception which is typical of C programming with arrays?

In the C programming world, you'd solve the situation merely by passing the number of items as an extra parameter, like so:

extern void x(int* y, int n);
void z(int n)
x(new int[n], n);
{{"}"}}

But in C++ there is a much better way to address this issue. It's called using a class and within that class have all the necessary details described as attributes inside that class. A better implementation of the new code to be written would be as such:

extern void x(int* y);
class SpecializedArray {{"{"}}
int* _array;
int _size;
public:
SpecializedArray(int n):
_array(new int[n]), _size(n) {{"{"}}
x(y);
{{"}"}}
~SpecializedArray() {{"{"}}
delete _array;
{{"}"}}
{{"}"}}

The above function x typically belongs to a piece of code inherited into the new application from either a third party library or an earlier project and therefore cannot be changed. C++ specializes in allowing the new application to centralize this issue and add the intended behaviour to the old code without jeopardizing working code that has already been supplied. At some point in the future, if it was determined that only 500 array elements can be fitted into the function called x, then the new class can be fitted at runtime to make a check for this without having to change the old code:

SpecializedArray(n):
_array(new int[n]), _size(n) {{"{"}}
if ( n>500 )
throw new OutofRangeException();
x(y);
{{"}"}}

In this way, you are checking at run-time for possible problems with the code instead at compile time which may not be an option in certain unknown circumstances.

#7: Test your code in advance

In practice, there are two types of error correction in the real world:

  1. Corrective Maintenance
  2. Preventative Maintenance

Corrective Maintenance is the use of low level debuggers and compile time print statements to try and catch a problem manifesting itself in your application and it's not a practical solution for production environments.

Preventative Maintenance is the use of formal tests and formal programming practices to prevent errors from forming in the first place.

Another approach to catching run-time errors is to insert various run-time checks with special constructs like you might see in the STL library or some third party library like the new GSL framework out of Microsoft. If you have to use these libraries to catch subtle errors in your code you are making work for yourself. Take the time to test all the values going into your model first and invest time to make sure no value appears during the production version of your code that has not been simulated in advance.

There is a popular programming practice today known as Test Driven Development or TDD for short. With TDD, you take the time to make the time to write your tests in advance. Then you write your production code to meet the test suite that has been written. Ideally, you have more than one programmer available to work in the project. One programmer consults with the requirements analysis people to determine what the functionality for the system is and writes tests to match the functionality expected from the system. In TDD, tests are written in such a way as to cause the production code to fail unless the most finite detail of the test case are resolved. The programmer assigned to write the production code then fills in the functionality as described in the test suite. In this way, you can formally eliminate a much greater percentages of time lost tracking subtle problems in production code as well as save a lot of time not writing code thats not needed. You also must have a way of testing your code upon demand and this encourages a MVC or Model-View-Control paradigm in your software.

Programmers new to C++ often over look this disciplined approach to writing code and end up spending a lot of time tracking down a lot of subtle problems which results in costly delays in project delivery times.

With an elaborate test suite formed in advance, a programmer now has a powerful and automated way of resolving problems with the production code merely by spending the time setting up a test for every known situation that the production code will encounter. It simply does not matter how the programmer resolves the issue, even at the last minute. The focus is on the test code, not the production code, hence there is nothing in the production code that has not been tested in advance. Hence, very little need for low level debuggers in a production environment.

#8: Waste not, want not

In the world of C programming, you had to manually deallocate memory. With the new C++ constructor / destructor method of doing things, memory allocation became far more streamlined. You still had to take the time to write specialized constructors their and accompanying destructors, but the usage of memory resources after than become almost automatic. In fact, when it came to heap memory allocation you don't even need to directly call the destructor.

const int Max = 50;
class Account {{"{"}}
std::string name;
float balance;
float* debits;
float* credits;
public:
Account(std::string& name, float balance)
:name(name), balance(balance) {{"{"}}
debits = new float[Max];
credits = new float[Max];
{{"}"}}
~Account() {{"{"}}
delete [] debits;
delete [] credits;
{{"}"}}
{{"}"}};
int main(int argc, const char * argv[]) {{"{"}}
Account* samsAccount = new Account("Sam",500.00);
delete samsAccount;
{{"}"}}

The above class, while simplistic, is a clear demonstration of how much more organized your code is with C++ over C. The following would be the C implementation of the above.

const int Max = 50;
struct Account {{"{"}}
const char* name;
float balance;
float* debits;
float* credits;
{{"}"}};
struct Account* createAccount(const char* name, float balance) {{"{"}}
Account* account = (Account*)malloc(sizeof(Account));
account->name = (char*)malloc(strlen(name)*sizeof(char));
memcpy(&account->name,name,Max);
account->balance = balance;
account->debits = (float*)malloc(Max*sizeof(float));
account->credits = (float*)malloc(Max*sizeof(float));
return account;
{{"}"}}
void deleteAccount(struct Account* account) {{"{"}}
free(&account->name);
free(account->debits);
free(account->credits);
free(account);
{{"}"}}
int main(int argc, const char * argv[]) {{"{"}}
struct Account samsAccount = createAccount("Sam",500.00);
deleteAccount(samsAccount);
{{"}"}}

What you see above is a very, very disciplined approach to writing code to remove allocated resources in a C language environment. As you can see, an excessive amount of extra code is needed to achieve the same results and the preceding C++ code and this was just for demonstration purposes. In practice, C++ simplifies the freeing up of resources compared to similar approaches taken in C by a considerable margin.

Further, what you don't realize is that with C++, you do not have to write destructors for code you inherit into your application. Take for example the attention to detail to copy the name of the person using the C language approach. Before the name can be copied into the Account structure, memory has to allocated for the Account itself, then memory for the name of the account has to allocated into the Account, then the contents of the name passed as a parameter has to be copied into the newly created memory for the name inside the Account. And this process has to be repeated and meticulously managed throughout the entire development process of the application which includes changes to already working code. Because C++ makes use of destructors the C++ version of this functionality is minimal, simply assign the STL attribute for the name with the parameter and constructors and destructors are automatically run for you.

#9: Less code is more code

The first time I heard that expression it was inside the beginning pages of The C++ Programming Language and it makes total sense. Further, it has been my experience that the more code you write the more code you have to write to make it flexible. Far too often novice programmers will think that they have to do things like write a copy constructor and then override the equals assignment operator and then make the two of them interchangeable "just in case". This too is where writing your test cases ahead of time reduces the need to write extra code that is really not necessary in order to achieve the end product. With C++, it has been my experience that figuring out what the code is suppose to do in the first place is far, far, far more important than trying to write a lot of unnecessary code "just in case" some future upgrade or complication might benefit from it.

#10: Parameters should be immutable

In the C language world, it was often beneficial to change the array passed as a parameter to a function as opposed to creating a new array to be passed back at the end of the function. While this seemed like a logical way of reducing the amount of code you would have to write, it soon led to a lot of complications as you had to often take a double take on any given function to find out if the original parameter being passed was being modified.

In the C++ language world this changed considerably. Such that it became far more easier to simply duplicate the original array and return the modified copy thanks in part to the C++ means of constructors and destructors automatically creating the necessary resources for you with no explicit calls to do so. It also made for easier to manage code.

In a ideal world, a function begins at the start with immutable parameters, makes whatever changes it needs to a duplicate of the parameters and then returns the changed results at the end of the function. This three step mentality towards writing code makes code writing much more easy to do and easy to troubleshoot.

#11: Don't reinvent the wheel

The Structured Template Library came into being because many standard design patterns were being repeated inside production code differing only in application specifics. In the beginning computer science students had to learn how to put together a link list structure in memory from scratch and several times for a single project. This soon led to a lot of repetition in code for basic operations such as vectors and hash maps. Learning how to utilize a library like STL greatly reduces the need to reinvent the wheel as the only thing that really changes in the new application is the actual application data and not how the application data is maintained in memory.

There are also many libraries inside the C++ framework to deal with various resources common with the operating system as well as many inherited from the C programming world. Familiarity with these previous libraries generally saves a lot in time and investment on the part of the application programmer.
