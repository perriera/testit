Evolution of C++

The need for reusable code brought about the need for C++ just as the need for portable code brought about the need for C in an industry where time is money.

Solid State & Vacuum Tubes

In the beginning you programmed a computer literally by setting a series of switches on the computer, which was still a step above programming a computer using a series of vacuum tubes, no different than the back of an old fashioned black and white television set:

Vacuum tube computers are programmable digital computers using vacuum tube logic circuitry. They were preceded by systems using electromechanical relays and followed by systems built from discrete transistors.

In 1943 the world's first programmable computer took on the name Colossus and was used to break German Lorenz cipher. It was based on vacuum tubes. Vacuum tube computers were the basic for digital computers as a basis for it's logic circuitry. Before vacuum tubes engineers devised a computers using electromechanical relays.

Discrete Transistors

How we take it for granted today, but back in the day, a simple tripod transistor began what we know today as the 8008, a forerunner to the late great 8080 which paved the way for a company we know as Intel today. Other companies produced similar technology such as the late great 6502 and the 6809, both very popular processors in the 80's.

The transistor allowed engineers to now piece together what we know today as the CPU or Central Processing Unit. Other types of computer chips such as memory and chips that interfaced with the real world were also created based upon the transistor.

Central Processor Unit (CPU)

The first CPU was very basic by today's standards but in many ways it was like stepping on the moon for the very first time. John Kennedy played a major role in the development of the transistor and in the development of the CPU such that when he ordered the American government to pursue putting a man on the moon, all this technology came into being by way of necessity.

What you see above is the very popular 8080A CPU as developed and sold by Intel. It was used in the first IBM PC along with Bill Gate's Microsoft Disk Operating System known then as MS DOS.

John Kennedy and The Space Race

It was the Russians that actually put a satellite into orbit first (Sputnik). Alarmed at this technological development, the American government took on the impossible task of putting a man on the moon. A by product of all the technology required to do that gave us the transistor and from their the Americans were able to devise the world's first CPU.

Had it not been for the Space Race, we simply would not have the technology as it exists today. Moreover, the CPU could be considered one of the greatest milestones towards establishing better relations with the Russians as the Russians soon found out that Silicon Valley was able to make CPUs a lot cheaper, faster and better than Moscow could. In this way, the capitalistic system of the west outran the communist system of the east. Hence, Kennedy is bit of an unsung hero when it comes to establishing peaceful relations with the former communist block.

It is with a bit of irony however, that the world's first orbital satellite, the Sputnik 1 took on the same basic shape as the world's first commercial transistor.

It brings back the age old question: "Which came first the chicken or the egg?" or in this case: "What came first the transistor or the satellite?"

Machine Language

The next big break through was the ability to program a computer using numeric codes. Some codes were instructions to the processor, others were data to be acted upon. What you see below is literally 'Machine Language', that is, a language that the CPU can understand natively. It was very fast but also very cryptic and unique to each type of CPU and thus not very portable.

But at least you had a way of programming a machine without having to swap in and out vacuum tubes or manually adjust a series of switches. In this way, the early software industry rejoiced over the prospect of being able to program a computer using a keyboard instead of punch cards.

Assembler Language

Needless to say, looking at hexadecimal numbers all day became monotonous, worse still, getting a single digit wrong and the CPU would simply lock up (or try to burn itself out), often only being able to be reset by literally 'pulling the plug' out of the wall. Hence the need for some method of organizing code using the alphabet became an absolute necessity.

section .text
global _start ;must be declared for linker (ld)
_start: ;tell linker entry point
mov edx,len ;message length
mov ecx,msg ;message to write
mov ebx,1 ;file descriptor (stdout)
mov eax,4 ;system call number (sys_write)
int 0x80 ;call kernel
mov eax,1 ;system call number (sys_exit)
int 0x80 ;call kernel
section .data
msg db 'Hello, world!',0xa ;our dear string
len equ $ - msg ;length of our dear string

Assembler was not the prettiest language to do stuff in but it was very, very fast and a lot better to program in than pure numbers and could be documented. Sadly, each Assembler language was specific to each brand of CPU and memory type.

Hence you would describe the program you wanted to run on the computer using something that resembled very cryptic English (especially TLAs or Three Letter Acronyms), at least at the basic level, that would be 'assembled' into Machine Language. You could also document your code using English (or French, etc.) which was a huge step towards writing maintainable production code. While not a huge leap over pure Machine Language, Assembler Language gave you a much better idea as to where the troublesome code might be found.

The Simula Language

With the popularity of C came an incredible amount of code developed that could be ported to different computer hardware and their operating systems. Developers soon found out however that the more code they wrote, the more code they had to write to make changes to it. Hence the need to write code in such a way as to make use of what was already written without changing it, except in certain circumstances appeared.

Begin
Class Glyph;
Virtual: Procedure print Is Procedure print;
Begin
End;
Glyph Class Char (c);
Character c;
Begin
Procedure print;
OutChar(c);
Route walk(); // don't

Ref (Glyph) Array elements;

Procedure print;
Begin
Integer i;
For i:= 1 Step 1 Until UpperBound (elements, 1) Do
elements (i).print;

rgs (1):- New Char ('A');
rgs (2):- New Char ('b');
rgs (3):- New Char ('b');

rg:- New Line (rgs);
rg.print;
End;

Simula 67 introduced objects, classes, inheritance and subclasses, virtual procedures, coroutines and discrete event simulation and features garbage collection. Also other forms of subtyping (besides inheriting subclasses) were introduced in Simula derivatives. Sadly, it was neither fast nor portable and generally restricted for use on large mainframes in university settings.

The 'certain circumstances' being utilizing the old C code for newer applications while retaining the old C code's functionality for it's original purpose. Most students graduating out of the universities only knew how to program in C, only a handful of programmers that had been introduced to an experimental language known as Simula 67, where code was organized into a series of classes where data could be hidden and inherited from seen an opportunity to mix this capability with the C language.

The C Language

While Assembler was a big break from Machine Language, it was still unique to each type of CPU and each type of CPU varied from CPU manufacturer as well with each new generation of CPU developed. But what programmers working for different CPU manufactures noticed was that they were all generally building the same software to run on their vendor specific hardware.

#include <stdio.h>
int main(int argc, char* argv[])
{{"{"}}
printf("Hello, world!\n");
{{"}"}}

C brought direct mapping of hardware features, was very efficient (small & fast), had a large code base and was very portable between hardware and cpu vendors.

Hence the need to have some sort of language that could be ported between vendors and even between older and newer CPUs made itself readily apparent. Hence the inception of The C Programming Language by Dennis Ritchie between the years of 1969 and 1973 at Bell Labs.

With it's relatively verbose, hardware platform independent and more english like coding syntax, Assembler Language programmers could now write C code for one vendor or CPU and have it ported to compatible C compilers on other hardware from other vendors. Where the C compilers for other vendors would covert the C code into the Assembler Language unique for their CPU.

The C++ Language

C++ started off as 'C with Classes' which essentially introduced programmers to a whole new way of organizing code in terms of classes, constructors, destructors, encapsulation, inheritance, virtual methods, operator overloading and exceptions to name a few concepts to address common Design Patterns that were making themselves apparent in large bodies of C code.

#include <iostream>
int main(int argc, char* argv[])
{{"{"}}
std::cout << "Hello, world!\n";
{{"}"}}

C++ brought classes, constructors, destructors, objects, inheritance, encapsulation, polymorphism, templates, etc, making life a lot easier.

To the inexperienced, one way of getting previously written code to work in a new application was merely to setup what they call a 'flag' using an 'if statement' deep inside the code. This technique would work great the first two or three times. After that, The Law of Diminishing Returns took over and productivity typically slowed to a stand still accordingly as either changing working code often ruined it for it's original purpose or duplicating the original code for the new purpose did away with any real concept of utilizing a single code base producing more sets of nearly identical code to maintain.

With the appearance of C++, programmers had a much better way of utilizing old C code without changing it for newer applications. This is essentially why C++ became so popular and why Bjarne Stroustrup is generally regarded as a computer language pioneer in the industry. Also, being able now to build code from an established code base paved the way to realistically setup test code to verify the code's functionality given that you were now spending more time writing new code and not essentially re-writing old code. The Java Language

Given that C++ was being developed in the early 80's, future languages such as C#, D, Java, Ruby took the lessons learned in C++ and implemented their own rendition of those features into those languages. For example, the parallels between C++ and Java are almost one to one with most of the object-oriented techniques developed and implemented in C++.

import java.lang.*;
public class HelloWorld {{"{"}}
public static void main(String[] args)
{{"{"}} System.out.println("Hello, world!"); {{"}"}}
{{"}"}}

Java came with a built in garbage collector, the key fundamental difference between it and C++.

Some would try to argue that the newer languages are better than it's predecessor for specific purposes but that would only matter with respect to what you had in mind for a particular application. Typically, anything that requires direct management of resources such as memory or off line storage (such as databases) have a better foundation with C++. For example, most of the web runs on servers that utilize either Apache, Linux and NodeJS applications, all of which are platforms founded upon C and C++. That being said, if you are not developing operating system level tools for use by business applications you are typically far better off utilizing a language like Java that handles memory management for you as well as enforce stronger type checking on syntax, ergo, resulting in fewer classical programming mistakes before they have a chance to manifest themselves.

But the general rule of thumb is that if you need speed and control you use C++, if you want automated garbage collection and fewer subtle problems go for Java. In fairness Java, in the hands of an expert, can run nearly as fast as C++ but there is no guarantee that it's automated memory management will work consistently. Before the days of writing formal test suites programmers had to rely mostly on debugging techniques and sometimes hardware level debuggers because most programming shops simply didn't have the time to test their code properly from a business cost perspective. Ergo, performing hardware level debugging in a production environment to track down either a null pointer assignment, a problem with the hardware itself or an issue with a telecommunications protocol is an experience you don't soon forget. Hence the benefits of using the right tool for the right job. The Swift Language

During the 90's there were two major contenders for object-oriented software design based upon the language of C. They were C++ and Objective-C. Objective-C became very popular on the MAC platform whereas C++ thrived in the PC, UNIX and Linux environments. In the end the Objective-C language would evolve into the Swift language that is now open source and an excellent language for mobile devices.

#import <Foundation/Foundation.h>
int main(int argc, char * argv[]) {{"{"}}
{{"{"}}
NSLog(@"Hello, world!\n");
{{"}"}}

For example, given the improvements made to Objective-C, the Swift language can express the above code in as simple as one single statement.

print("Hello World!")

Swift however is yet another language that contains it's own garbage collector and therefore does not provide the same level of machine control that C++ was designed for.

How C++ got it's name . . .

The term 'C++' was first used in December, 1983 and the credit for that goes to Rick Mascitti, who, while inquiring about the nature of the 'C with Classes' language may have used the expression 'C plus Classes'. Moreover, there is a construct originating from the C programming language called the incremental operator which simply increments a value by one each time it is called using the '+' symbol twice. This logical, albeit, impromptu nickname for the more formal 'C with Classes' title soon caught on and became the official name of C++.
